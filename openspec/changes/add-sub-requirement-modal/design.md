## Context

### 背景
当前系统已经实现了需求管理的基础功能，包括需求列表、详情面板、评论、操作日志等。父子需求的数据模型也已经建立（parent_id字段已添加到数据库），但缺少用户友好的子需求创建界面。

### 约束条件
1. **设计风格**：必须严格遵循Pencil设计Node ID: uiIIm的规范
2. **技术栈**：Next.js + React + TypeScript + Tailwind CSS
3. **数据库**：PostgreSQL，使用现有parent_id字段
4. **兼容性**：向后兼容，不影响现有功能

### 利益相关者
- 终端用户：需要直观创建子需求的界面
- 开发团队：需要可维护的代码实现

## Goals / Non-Goals

### Goals
- 实现与Pencil设计一致的子需求创建弹窗
- 支持完整的子需求字段填写
- 使用真实数据（数据库持久化）
- 确保良好的用户体验

### Non-Goals
- 不实现批量创建子需求
- 不实现子需求模板功能
- 不实现子需求导入导出
- 不修改现有的需求列表展示逻辑

## Decisions

### 决策1：弹窗 vs 右侧面板
**选择**：使用弹窗（Modal）而非右侧面板

**理由**：
1. Pencil设计明确使用弹窗形式
2. 子需求创建是独立的操作，不需要与父需求详情同时查看
3. 弹窗可以更聚焦在创建任务上
4. 与右侧面板的编辑模式形成区分

**替代方案**：
- 在右侧面板内嵌创建表单：会压缩空间，不符合设计
- 跳转新页面：破坏上下文，用户体验差

### 决策2：工作项ID生成策略
**选择**：基于父需求ID添加层级后缀

**规则**：
- 父需求：REQ-001
- 一级子需求：REQ-001-1, REQ-001-2...
- 二级子需求：REQ-001-1-1, REQ-001-1-2...

**理由**：
1. 直观展示层级关系
2. 避免ID冲突
3. 便于用户理解
4. 符合工作项ID的命名规范

**实现方式**：
```typescript
function generateSubRequirementId(parentWorkItemId: string, existingIds: string[]): string {
  const baseId = parentWorkItemId;
  let index = 1;
  let newId = `${baseId}-${index}`;
  
  while (existingIds.includes(newId)) {
    index++;
    newId = `${baseId}-${index}`;
  }
  
  return newId;
}
```

### 决策3：字段默认值策略
**选择**：继承父需求的部分字段作为默认值

**策略**：
- 优先级：继承父需求优先级
- 项目ID：继承父需求项目ID
- 状态：默认"todo"
- 负责人：默认为空
- 积分：默认为0或根据优先级自动计算

**理由**：
1. 减少用户输入
2. 保持父子需求的一致性
3. 用户仍可修改

### 决策4：错误处理方式
**选择**：前端实时验证 + 后端二次验证

**策略**：
1. 前端：字段失焦时验证，显示即时反馈
2. 后端：API层完整验证，返回详细错误信息
3. 错误提示：中文（根据用户语言设置）

**理由**：
1. 提升用户体验（即时反馈）
2. 确保数据安全（后端验证）
3. 易于维护（统一错误格式）

## Risks / Trade-offs

### 风险1：工作项ID长度
**风险**：深层级子需求ID可能过长（如REQ-001-1-2-3-4）

**缓解**：
- 数据库字段长度足够（VARCHAR(100)）
- 实际业务中深层级需求较少
- 考虑后续优化ID生成策略（如使用UUID后缀）

### 风险2：父需求删除级联
**风险**：删除父需求会级联删除所有子需求

**缓解**：
- 数据库已设置ON DELETE CASCADE
- 删除前显示确认对话框
- 操作日志记录删除操作

### 风险3：并发创建ID冲突
**风险**：多个用户同时创建子需求可能导致ID冲突

**缓解**：
- 使用数据库唯一约束
- 后端重试机制
- 异常情况返回友好提示

2. 修改SubRequirementList组件，添加触发逻辑

4. 添加国际化翻译

### 回滚方案
1. 移除SubRequirementModal组件
2. 恢复SubRequirementList组件
3. 不影响已有数据

## Open Questions

   **A**: 当前实现支持，但实际业务中建议使用不超过二级

2. **Q**: 子需求是否可以再有自己的子需求？
   **A**: 技术上支持，业务上根据需求决定

3. **Q**: 是否需要子需求模板功能？
   **A**: 不在当前变更范围内，后续可考虑

4. **Q**: 如何处理父需求状态变更时子需求的状态？
   **A**: 当前不自动同步，保持独立状态管理
