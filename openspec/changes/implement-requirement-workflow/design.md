## Context

本变更基于产品 PRD「需求管理-20260221」实现完整的需求管理工作流。当前系统已有基础的需求 CRUD 功能，但需要升级为支持完整敏捷工作流的解决方案。

**约束条件**:
- 使用 PostgreSQL 作为数据存储（根据 project.md，已配置 POSTGRES_URL）
- Next.js 15 App Router 架构
- Tailwind CSS 样式系统
- 支持中文/英文/日文国际化
- UI 需参考 Pencil Node YNcpT（深色导航栏风格）
- 需配合 `compact-ui-sizing` 变更使用紧凑尺寸

**利益相关者**:
- 终端用户：需要高效管理需求的项目管理者和开发者
- 开发团队：需要维护可扩展的代码架构

## Goals / Non-Goals

### Goals
- 提供完整的需求生命周期管理（创建-编辑-查看-流转）
- 支持父子需求的无限层级结构
- 实现需求与任务的双向关联
- 提供评论、附件协作功能
- 记录完整的操作审计日志
- 提供流畅的右侧滑出面板交互体验

### Non-Goals
- 不实现复杂的权限控制（RBAC）- 使用现有简单认证
- 不实现实时协作（如 WebSocket 同步）- 使用轮询或手动刷新
- 不实现邮件/通知推送 - 仅站内操作记录
- 不实现复杂的报表分析 - 仅基础列表和详情

## Decisions

### 决策 1：使用右侧滑出面板替代弹窗
**决策**: 采用右侧滑出面板（Slide-out Panel）作为新增/编辑/查看需求的统一交互方式。

**理由**:
- 保持用户在项目上下文中，无需切换页面
- 与现有 Drawer 组件风格一致
- 支持更丰富的内容展示（评论、操作记录等区块）
- 参考 PRD 设计和现代 SaaS 产品趋势（如 Linear、Notion）

**替代方案考虑**:
- 模态弹窗：空间受限，不适合展示复杂详情
- 独立页面：失去上下文，需要返回操作

### 决策 2：工作项 ID 生成策略
**决策**: 使用 `REQ-{sequence}` 格式，子需求使用 `父ID-{childSequence}` 层级后缀。

**理由**:
- 直观展示层级关系
- 便于人工识别和引用
- 符合 PRD 要求

**生成规则**:
```typescript
// 顶层需求
const workItemId = `REQ-${nextSequence.toString().padStart(3, '0')}`;

// 子需求
const workItemId = `${parentWorkItemId}-${childIndex + 1}`;
```

**潜在问题**:
- ID 可能过长（深层嵌套时）
- 需要确保全局唯一性

**缓解措施**:
- 数据库添加唯一索引约束
- 限制层级深度（软限制，不强制）

### 决策 3：父子需求状态聚合规则
**决策**: 父需求状态根据子需求状态自动聚合。

**聚合逻辑**:
```
IF 所有子需求 IN (已完成, 已验收, 已关闭):
  父需求状态 = 已完成
ELSE IF 任意子需求 = 进行中:
  父需求状态 = 进行中
ELSE IF 所有子需求 = 待办:
  父需求状态 = 待办
ELSE:
  保持当前状态（混合状态）
```

**理由**:
- 减少手动更新父需求状态的负担
- 自动反映子需求整体进度

**注意事项**:
- 仅在子需求状态变更时触发聚合
- 手动修改父需求状态时需提示可能影响聚合逻辑

### 决策 4：评论附件存储方案
**决策**: 附件先存储在本地文件系统（或配置的外部存储如 OSS/S3），数据库仅存储元数据。

**数据库表结构**:
```sql
CREATE TABLE attachments (
  id UUID PRIMARY KEY,
  name VARCHAR(255),
  url VARCHAR(500), -- 文件访问 URL
  size INTEGER,
  type VARCHAR(100),
  comment_id UUID REFERENCES comments(id)
);
```

**理由**:
- 数据库不存储大文件，保持性能
- 灵活支持多种存储后端
- 符合 PostgreSQL 最佳实践

**配置方式**:
- 开发环境：本地文件系统
- 生产环境：环境变量配置 OSS/S3

### 决策 5：操作记录实现方式
**决策**: 使用数据库触发器 + 应用层双保险记录操作日志。

**应用层记录**:
- 在 API 层显式调用 `createOperationLog()` 函数
- 优点：灵活，可记录业务语义（如「张三修改了状态」）

**数据库触发器（可选增强）**:
- 作为备份，记录所有字段级变更
- 优点：不遗漏任何变更

**理由**:
- 保证操作记录的完整性
- 满足审计要求

### 决策 6：需求列表虚拟滚动
**决策**: 首版不使用虚拟滚动，仅实现基础分页。

**理由**:
- 个人项目管理场景下需求数量通常 < 1000
- 首版追求快速交付
- 后续可基于性能数据评估是否需要虚拟滚动

**替代方案**:
- 如性能测试显示需要，后续可引入 `react-window` 或 `@tanstack/react-virtual`

## Risks / Trade-offs

### 风险 1：数据库 Schema 变更导致数据丢失
**风险**: Migration 脚本错误可能导致现有需求数据丢失。

**缓解措施**:
- 开发环境充分测试 migration
- 生产环境部署前备份数据库
- Migration 脚本使用事务包裹
- 增量式变更（先加字段，后删字段）

### 风险 2：父子需求层级过深影响性能
**风险**: 用户创建过深层级的需求（如 10+ 层）可能导致递归查询性能下降。

**缓解措施**:
- 列表查询使用非递归方式（先查所有，内存构建树）
- 软限制：UI 提示建议层级不超过 5 层
- 后续可考虑使用 PostgreSQL 递归 CTE 优化

### 风险 3：并发编辑冲突
**风险**: 多个用户同时编辑同一需求可能导致覆盖。

**缓解措施**:
- 首版使用简单乐观锁（最后写入者胜）
- 后续可考虑实现编辑锁或冲突检测

### 风险 4：附件存储成本
**风险**: 用户上传大量附件导致存储成本上升。

**缓解措施**:
- 单文件 20MB 限制
- 单条评论 5 个附件限制
- 定期清理孤儿附件（未关联评论的附件）

## Migration Plan

### 数据库 Migration

**Step 1**: 备份现有数据库
```bash
pg_dump $POSTGRES_URL > backup_$(date +%Y%m%d).sql
```

**Step 2**: 执行 Schema 变更
- 扩展 requirements 表字段
- 创建 comments、operation_logs、attachments 表

**Step 3**: 数据迁移
- 为现有需求生成 work_item_id（基于创建时间顺序）
- 初始化空的操作记录表

**Step 4**: 验证
- 检查所有表结构正确
- 验证现有数据完整性

### 代码部署

**Step 1**: 部署新 API 路由
- 先部署只读 API（列表、详情）
- 验证正常工作

**Step 2**: 部署新 UI 组件
- 并行部署，不影响旧功能

**Step 3**: 切换主页面
- 替换项目详情页的需求 Tab
- 保留旧看板视图代码（注释或备份）便于回滚

**Step 4**: 验证
- 进行完整功能测试
- 监控错误日志

### Rollback Plan

如发现问题可快速回滚：
1. 恢复旧版页面代码（requirement-kanban 视图）
2. 数据库变更不可逆，但旧代码兼容新 schema（新增字段可选）
